# مشخصات فنی (Spec) — سیستم مدیریت سفارش آنلاین کافه‌بستنی

نسخه: 2.0 — تاریخ: 2026-07-24 — وضعیت: در حال توسعه

---

## ۱. معرفی پروژه

سیستم سفارش‌گیری آنلاین برای یک کافه‌بستنی که آبمیوه، بستنی، دسر، شیرموز و قهوه می‌فروشد.

### جریان اصلی (Flow)

```
مشتری بیرون کافه ──اسکن QR منو──▶ صفحه منوی وب ──انتخاب آیتم‌ها──▶ ثبت سفارش
        │
        ▼
   دریافت QR کد مخصوص سفارش روی موبایل
        │
        ▼
مشتری داخل کافه ──نشون دادن QR──▶ صندوق‌دار با پنل لپ‌تاپ اسکن می‌کنه
        │
        ▼
جزئیات سفارش (اقلام، قیمت، جمع کل) روی لپ‌تاپ ──▶ آماده‌سازی ──▶ تحویل + پرداخت حضوری
```

### تصمیمات کلیدی (تاییدشده توسط کارفرما)

| موضوع | تصمیم |
|---|---|
| پرداخت | حضوری موقع تحویل (نقدی/کارتخوان) — بدون درگاه آنلاین |
| امنیت پنل‌ها | لاگین ساده با رمز عبور (پنل صندوق + پنل مدیریت) |
| چرخه وضعیت سفارش | کامل: `pending → preparing → ready → completed` + `cancelled` |
| دیتابیس | SQLite (فایل local، مناسب شروع) |
| زبان و جهت UI | فارسی، راست‌به‌چپ (RTL) |
| واحد پول | تومان |

## ۲. استک تکنولوژی

| لایه | تکنولوژی |
|---|---|
| Backend | Python 3.11+ / FastAPI / SQLAlchemy 2 / Pydantic v2 / Uvicorn |
| Database | SQLite |
| Frontend | React 18 + TypeScript + Vite |
| استایل | Tailwind CSS (RTL) |
| تولید QR سفارش | کتابخانه `qrcode` پایتون (خروجی PNG/SVG به‌صورت base64 در پاسخ API) |
| اسکن QR در مرورگر | کتابخانه `html5-qrcode` (استفاده از وب‌کم لپ‌تاپ در پنل صندوق) |
| احراز هویت | رمز عبور ثابت (در فایل config) → توکن JWT ساده برای پنل‌ها |

> نکته: QR کد منو (که بیرون کافه چاپ می‌شه) فقط یک URL ثابت به صفحه منو است و یک‌بار ساخته می‌شه — نیازی به تولید داینامیک ندارد. QR کد سفارش داینامیک است و بعد از ثبت هر سفارش تولید می‌شود.

## ۴. مدل‌های دیتابیس

### 4.1 Category (دسته‌بندی)

| فیلد | نوع | توضیح |
|---|---|---|
| id | INTEGER PK | |
| name | TEXT, unique, not null | مثل «آبمیوه»، «بستنی»، «دسر»، «شیرموز»، «قهوه» |
| display_order | INTEGER, default 0 | ترتیب نمایش در منو |
| is_active | BOOLEAN, default true | مخفی کردن کل دسته از منو |

### 4.2 Product (محصول)

| فیلد | نوع | توضیح |
|---|---|---|
| id | INTEGER PK | |
| category_id | FK → Category | |
| name | TEXT, not null | مثل «آب هویج»، «بستنی سنتی» |
| description | TEXT, nullable | توضیح کوتاه اختیاری |
| price | INTEGER, not null | قیمت به تومان (عدد صحیح، بدون اعشار) |
| is_available | BOOLEAN, default true | ناموجود کردن موقت بدون حذف |
| display_order | INTEGER, default 0 | ترتیب نمایش داخل دسته |
| created_at / updated_at | DATETIME | |

### 4.3 Order (سفارش)

| فیلد | نوع | توضیح |
|---|---|---|
| id | INTEGER PK | |
| code | TEXT, unique, not null | کد یکتای سفارش (UUID کوتاه) — محتوای QR کد |
| status | TEXT enum | `pending` / `preparing` / `ready` / `completed` / `cancelled` |
| source | TEXT enum | `online` (از موبایل مشتری) / `walk_in` (ثبت دستی صندوق) |
| customer_name | TEXT, nullable | اسم اختیاری برای صدا زدن مشتری |
| note | TEXT, nullable | توضیحات مشتری (مثلاً «بدون شکر») |
| total_amount | INTEGER, not null | جمع کل به تومان — موقع ثبت محاسبه و ذخیره می‌شود |
| created_at / updated_at | DATETIME | |

### 4.4 OrderItem (قلم سفارش)

| فیلد | نوع | توضیح |
|---|---|---|
| id | INTEGER PK | |
| order_id | FK → Order | |
| product_id | FK → Product | |
| product_name | TEXT, not null | **اسنپ‌شات** اسم محصول در لحظه ثبت |
| unit_price | INTEGER, not null | **اسنپ‌شات** قیمت واحد در لحظه ثبت |
| quantity | INTEGER, not null, >0 | تعداد |

> ⚠️ نکته مهم: اسم و قیمت محصول در OrderItem کپی (snapshot) می‌شود تا اگر بعداً مدیر قیمت را عوض کرد، سفارش‌های قبلی دست‌نخورده بمانند.

### چرخه وضعیت سفارش

```
pending ──▶ preparing ──▶ ready ──▶ completed
   │            │
   └────────────┴──▶ cancelled
```

- `pending`: مشتری آنلاین ثبت کرده، هنوز به کافه نیامده / اسکن نشده
- `preparing`: QR اسکن شد (یا سفارش دستی ثبت شد) و در حال آماده‌سازی است
- `ready`: آماده تحویل
- `completed`: تحویل داده شد و پول گرفته شد
- `cancelled`: لغو (فقط از pending یا preparing)

## ۵. API Endpoints

Base URL: `/api`

### 5.1 عمومی — منوی مشتری (بدون لاگین)

| متد | مسیر | توضیح |
|---|---|---|
| GET | `/api/menu` | کل منو: دسته‌های فعال + محصولات موجود (`is_active` و `is_available`) به ترتیب `display_order` |

### 5.2 عمومی — سفارش مشتری (بدون لاگین)

| متد | مسیر | توضیح |
|---|---|---|
| POST | `/api/orders` | ثبت سفارش. بدنه: `{items: [{product_id, quantity}], customer_name?, note?}`. قیمت‌ها **از دیتابیس سرور** خوانده می‌شوند (نه از کلاینت). پاسخ: سفارش کامل + `qr_image` (base64) |
| GET | `/api/orders/{code}` | مشاهده سفارش با کد (برای صفحه موفقیت مشتری + QR) |

### 5.3 محافظت‌شده — احراز هویت

| متد | مسیر | توضیح |
|---|---|---|
| POST | `/api/auth/login` | بدنه: `{password}` → پاسخ: `{token}` (JWT). رمز در `.env` تعریف می‌شود |

### 5.4 محافظت‌شده — پنل صندوق (نیاز به توکن)

| متد | مسیر | توضیح |
|---|---|---|
| GET | `/api/cashier/orders/{code}` | بازیابی سفارش با اسکن QR — اگر pending باشد خودکار به preparing می‌رود |
| GET | `/api/cashier/orders?status=&date=` | لیست سفارش‌های امروز با فیلتر وضعیت |
| POST | `/api/cashier/orders` | ثبت سفارش دستی (walk_in) — همان بدنه ثبت سفارش + مستقیم با وضعیت preparing |
| PATCH | `/api/cashier/orders/{id}/status` | تغییر وضعیت: `{status: "ready" | "completed" | "cancelled" | ...}` با اعتبارسنجی گذار مجاز |

### 5.5 محافظت‌شده — پنل مدیریت (نیاز به توکن)

| متد | مسیر | توضیح |
|---|---|---|
| GET | `/api/admin/categories` | همه دسته‌ها (حتی غیرفعال) |
| POST | `/api/admin/categories` | ایجاد دسته |
| PATCH | `/api/admin/categories/{id}` | ویرایش دسته (نام، ترتیب، فعال/غیرفعال) |
| DELETE | `/api/admin/categories/{id}` | حذف دسته (فقط اگر محصول نداشته باشد) |
| GET | `/api/admin/products` | همه محصولات (حتی ناموجود) |
| POST | `/api/admin/products` | ایجاد محصول |
| PATCH | `/api/admin/products/{id}` | ویرایش محصول (قیمت، نام، موجودی، ...) |
| DELETE | `/api/admin/products/{id}` | حذف محصول (اگر در سفارشی استفاده شده، به‌جای حذف `is_available=false` می‌شود) |

### اعمال فوری تغییرات منو

منوی مشتری قیمت‌ها را کش نمی‌کند: هر بار باز شدن صفحه، `GET /api/menu` تازه صدا زده می‌شود و علاوه بر آن هر ۳۰ ثانیه یک refetch سبک انجام می‌شود. چون قیمت نهایی سفارش هم سمت سرور از دیتابیس محاسبه می‌شود، هیچ‌وقت قیمت قدیمی ثبت نمی‌شود. (در آینده در صورت نیاز می‌توان WebSocket اضافه کرد — برای این مقیاس لازم نیست.)

## ۶. QR کدها

### QR منو (ثابت — بیرون کافه)
- فقط URL صفحه منو است: `http://<آدرس-سرور>/`
- یک‌بار با اسکریپت کوچک یا هر ابزار آنلاین ساخته و چاپ می‌شود.

### QR سفارش (داینامیک)
- بعد از ثبت سفارش، سرور با کتابخانه `qrcode` تصویری می‌سازد که محتوایش **کد یکتای سفارش** است (مثل `A1B2C3D4`).
- تصویر به‌صورت base64 در پاسخ API برمی‌گردد و در صفحه موفقیت به مشتری نمایش داده می‌شود + کد متنی زیرش (برای حالتی که اسکن نشد، صندوق‌دار دستی تایپ کند).
- در پنل صندوق، `html5-qrcode` با وب‌کم لپ‌تاپ QR را می‌خواند، کد استخراج و `GET /api/cashier/orders/{code}` صدا زده می‌شود.

## ۷. Requirements غیرعملکردی

- **RTL و فارسی**: کل UI فارسی با فونت مناسب (Vazirmatn)، اعداد و قیمت‌ها با جداکننده هزارگان فارسی.
- **موبایل‌فرست**: صفحه منوی مشتری باید روی موبایل عالی باشد؛ پنل صندوق و مدیریت برای لپ‌تاپ.
- **اعتبارسنجی سمت سرور**: quantity مثبت، product موجود و available، گذار وضعیت مجاز.
- **دقت مالی**: قیمت‌ها عدد صحیح تومان — بدون float.
- **سادگی استقرار**: اجرا با دو دستور (`uvicorn` برای بک‌اند، `vite` برای فرانت). دیتابیس SQLite خودکار ساخته و seed اولیه منو اجرا می‌شود.
- **کد تمیز**: کامنت فارسی در بخش‌های مهم، جداسازی کامل frontend/backend.

## ۹. ویژگی‌های جدید (v2.0)

### ۹.۱ امنیت و احراز هویت
- **رمزهای جداگانه**: پنل صندوق (`CASHIER_PASSWORD`) و پنل مدیریت (`ADMIN_PASSWORD`) رمز جداگانه دارند
- **نقش توکن**: توکن JWT شامل فیلد `role` است (cashier یا admin)
- **محدودیت نرخ**: Rate limiting روی login — حداکثر ۵ تلاش در دقیقه
- **دسترسی نقش‌محور**: صندوق‌دار به پنل مدیریت دسترسی ندارد

### ۹.۲ حالت تعطیل کافه
- فلگ `IS_OPEN` در تنظیمات
- endpoint `PATCH /api/admin/settings/toggle` برای toggle وضعیت
- اگر کافه بسته باشد، `GET /api/menu` خطای 503 برمی‌گرداند
- صفحه منوی مشتری پیام زیبای فارسی «کافه بسته است» نمایش می‌دهد

### ۹.۳ WebSocket اعلان سفارش جدید
- endpoint `GET /ws/orders` برای اتصال WebSocket
- هنگام ثبت سفارش آنلاین، پیام `new_order` به همه کلاینت‌ها broadcast می‌شود
- در پنل صندوق: رفرش خودکار لیست + پخش صدای اعلان + toast notification

### ۹.۴ روش پرداخت
- فیلد `payment_method` در مدل Order (نقدی/کارتخوان)
- هنگام تکمیل سفارش، مودال انتخاب روش پرداخت نمایش داده می‌شود
- روش پرداخت در رسید چاپی و داشبورد نمایش داده می‌شود

### ۹.۵ شماره صف
- فیلد `queue_number` در مدل Order
- شماره صف روزانه از ۱ شروع می‌شود
- در صفحه موفقیت مشتری با فونت بزرگ نارنجی نمایش داده می‌شود

### ۹.۶ Idempotency
- فیلد `idempotency_key` اختیарی در ثبت سفارش
- اگر کلاینت همان key را دوباره بفرستد، سفارش قبلی برگردانده می‌شود

### ۹.۷ Auto-expire سفارش‌ها
- با APScheduler هر ۱۵ دقیقه بررسی می‌شود
- سفارش‌های pending بیش از ۶۰ دقیقه خودکار لغو می‌شوند

### ۹.۸ بکاپ خودکار شبانه
- هر شب ساعت ۲ بامداد فایل دیتابیس کپی می‌شود
- فقط ۷ فایل آخر نگه‌داشته می‌شود

### ۹.۹ داشبورد آمار
- صفحه `/admin/dashboard` با نمودار درآمد ۷ روز اخیر
- ۵ محصول پرفروش + جدول سفارش‌های اخیر
- کارت‌های آمار: درآمد امروز / تعداد سفارش / میانگین مبلغ

### ۹.۱۰ PWA
- manifest.json برای نصب به‌عنوان اپلیکیشن
- service-worker.js برای کش کردن درخواست منو (نمایش آفلاین)

### ۹.۱۱ ریسایز خودکار عکس
- آپلود عکس محصول با Pillow ریسایز به 400x400
- خروجی همیشه JPEG با کیفیت 85%

### ۹.۱۲ Index‌های دیتابیس
- `ix_order_status` روی فیلد status
- `ix_order_created_at` روی فیلد created_at
- `ix_order_code` unique index روی کد سفارش

### ۹.۱۳ مدیریت خطا
- `ErrorBoundary` برای catch خطاهای React
- `useApi` hook با retry خودکار و exponential backoff
